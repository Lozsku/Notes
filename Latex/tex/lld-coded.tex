\subsection{1. LRU Cache (+ LFU Variant Note)}\label{1-lru-cache--lfu-variant-note}

\subsubsection{Requirements}\label{requirements}

\textbf{Functional}

\begin{itemize}
\tightlist
\item
  \texttt{get(key)} --- return value or -1 if not present.
\item
  \texttt{put(key,\ value)} --- insert; if capacity exceeded, evict \textbf{least recently used} entry.
\item
  Both operations must be \textbf{O(1)}.
\end{itemize}

\textbf{Clarifying questions to ask}

\begin{itemize}
\tightlist
\item
  Is capacity fixed at construction? (Yes.)
\item
  Thread-safe? (Assume single-threaded unless asked; mention \texttt{threading.Lock} for MT.)
\item
  Key/value types? (Generics / \texttt{Any} for flexibility.)
\item
  On \texttt{put} of existing key, update value and recency? (Yes.)
\end{itemize}

\begin{itemize}
\tightlist

\item
  Mention: for thread-safety you\textquotesingle d wrap with a \texttt{threading.Lock} or use \texttt{collections.OrderedDict} (which Python\textquotesingle s stdlib provides for a simpler implementation, but interviewers want the manual version to test understanding).
\end{itemize}

\subsubsection{Extensibility discussion}\label{extensibility-discussion}

\begin{itemize}
\tightlist
\item
  \textbf{TTL / expiry:} Add \texttt{expiry:\ Optional{[}float{]}} to \texttt{DLinkedNode}. On \texttt{get}, check \texttt{time.time()\ \textgreater{}\ expiry} and treat as miss + remove. A background thread or lazy expiry on access both work.
\item
  \textbf{Thread-safety:} Wrap \texttt{get}/\texttt{put} with \texttt{threading.Lock}. For higher concurrency, shard the cache into N sub-caches keyed by \texttt{hash(key)\ \%\ N}.
\item
  \textbf{Generics:} Replace \texttt{int} with \texttt{TypeVar(\textquotesingle{}K\textquotesingle{})} / \texttt{TypeVar(\textquotesingle{}V\textquotesingle{})} and reuse across types.
\end{itemize}

\textbf{Interview takeaway: The sentinel head/tail trick is the key insight. Without sentinels you need \texttt{if\ node.prev} null-checks everywhere --- messy and error-prone. Name it explicitly.}

\subsection{2. Parking Lot}\label{2-parking-lot}

\subsubsection{Extensibility discussion}\label{extensibility-discussion-3}

\begin{itemize}
\tightlist
\item
  \textbf{New split type (shares-based):} Add \texttt{SharesSplit(SplitStrategy)} where each participant has N shares; their fraction = N\_i / sum(N). Zero changes to \texttt{Group}.
\item
  \textbf{Multi-currency:} Add \texttt{currency:\ str} to \texttt{Expense}; convert to a base currency using an injected \texttt{CurrencyConverter} before updating balances.
\item
  \textbf{Settle-up:} \texttt{settle(payer,\ payee,\ amount)} is just \texttt{add\_expense("Settlement",\ amount,\ payer,\ {[}payer,\ payee{]},\ ExactSplit(),\ exact\_shares=\{payer.user\_id:0,\ payee.user\_id:amount\})}.
\item
  \textbf{Persistence:} Replace \texttt{self.\_net} updates with DB writes. The \texttt{Group} class becomes a service layer calling a repository.
\end{itemize}

\textbf{Interview takeaway: The debt-simplification algorithm is a classic greedy: compute net positions, then greedily match biggest creditor with biggest debtor. It\textquotesingle s O(N log N) and minimizes the number of transactions (though not necessarily the amounts).}

  External requests: \texttt{request\_floor(floor,\ direction)} --- someone on floor 3 wants to go UP.
\item
  Internal requests: \texttt{select\_floor(elevator\_id,\ floor)} --- someone inside presses a button.
\item
  Scheduler assigns best elevator to each external request.
\item
  Each elevator has states: IDLE, MOVING\_UP, MOVING\_DOWN, MAINTENANCE.
\end{itemize}

\item

\item
  Priority floors (emergency)? (Out of scope for now.)
\item
  Which scheduling algorithm? (SCAN / "look" algorithm --- service in current direction first.)
\item
  Real-time simulation or just state transitions? (State transitions.)
\end{itemize}

\begin{itemize}
\tightlist

  \textbf{New scheduler (zone-based):} Implement \texttt{ZoneScheduler(Scheduler)} that restricts each elevator to a floor range. Inject at construction.

\end{itemize}

\subsection{6. Tic-Tac-Toe (Clean OO Board Game)}\label{6-tic-tac-toe-clean-oo-board-game}

\subsubsection{Requirements}\label{requirements-5}

\textbf{Functional}

\begin{itemize}
\tightlist
\item
  Two players, configurable board size (default 3\(\times\)3).
\item
  \texttt{make\_move(player,\ row,\ col)} --- place mark; validate; check win/draw.
\item
  Win condition: any row, column, or diagonal fully owned by one player.
\item
  Draw: board full, no winner.
\item
  Support pluggable players (human, AI/random).
\end{itemize}

\textbf{Clarifying questions to ask}

\begin{itemize}
\tightlist
\item
  Board size always 3\(\times\)3 or configurable? (Configurable; win requires N-in-a-row.)
\item
  Two players only, or more? (Two for now; mention extension.)
\item
  AI player? (Optional; use Strategy.)
\item
  Can the same player go twice? (No --- enforce turn order.)
\item
  Replay / undo moves? (Out of scope; mention Command pattern for undo.)
\end{itemize}

\subsubsection{Design patterns used}\label{design-patterns-used-5}

\begin{itemize}
\tightlist
\item
  \textbf{Strategy} --- \texttt{Player.choose\_move} makes each player type a strategy for move selection.
\item
  \textbf{Template Method} --- \texttt{Game.play()} defines the fixed turn loop; player-specific behavior is deferred to \texttt{choose\_move}.
\item
  \textbf{Single Responsibility} --- \texttt{WinChecker} is separate from \texttt{Board} (board manages state; checker evaluates it).
\item
  \textbf{Command} (mention) --- For undo, wrap each move in a \texttt{Command} object with \texttt{execute}/\texttt{undo}.
\end{itemize}

\subsubsection{Extensibility discussion}\label{extensibility-discussion-5}

\begin{itemize}
\tightlist
\item
  \textbf{N-in-a-row win (Connect-4 style):} Parameterize \texttt{WinChecker} with \texttt{win\_length\ \textless{}\ size}. Only the checker changes.
\item
  \textbf{More than 2 players:} Change \texttt{self.players} to a list; cycle with \texttt{(current\_idx\ +\ 1)\ \%\ len(players)}. \texttt{Board} already supports arbitrary marks.
\item
  \textbf{AI with Minimax:} Replace \texttt{RandomAIPlayer.choose\_move} with a minimax search tree. The \texttt{Player} interface is unchanged.
\item
  \textbf{Undo last move:} Wrap \texttt{board.place} in a \texttt{MoveCommand(Command)}. \texttt{game.undo()} calls \texttt{command.undo()} which clears the cell and decrements \texttt{\_moves\_made}.
\item
  \textbf{Networked game:} \texttt{RemotePlayer(Player)} overrides \texttt{choose\_move} to send the board state over a socket and wait for a response.
\end{itemize}

\textbf{Interview takeaway: The WinChecker separation is a key OOP point --- "Board manages state; WinChecker evaluates it." This satisfies SRP and makes unit testing trivial. Always mention the injectable moves iterator for testing HumanPlayer without stdin.}

\subsection{Common Patterns Across LLD Problems}\label{common-patterns-across-lld-problems}

\subsubsection{Entity-extraction checklist}\label{entity-extraction-checklist}

When you get a new problem, run through this in 5 minutes:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  \textbf{Nouns \(\rightarrow\) candidate classes:} Underline every noun. Parking Lot \(\rightarrow\) ParkingLot, Level, Spot, Vehicle, Ticket, Fee.
\item
  \textbf{Verbs \(\rightarrow\) candidate methods:} Circle every verb. "find available spot" \(\rightarrow\) \texttt{find\_spot(vehicle)}.
\item
  \textbf{Prune attributes:} Nouns that are just data fields (name, price, timestamp) become fields, not classes.
\item
  \textbf{Find the variability:} What changes? Pricing? Split type? Scheduling algorithm? That variability \(\rightarrow\) Strategy pattern.
\item
  \textbf{Find the lifecycle:} What has state? Elevator (Idle \(\rightarrow\) Moving), Ticket (Open \(\rightarrow\) Closed) \(\rightarrow\) State pattern or enum.
\item
  \textbf{Find the singleton:} What is there exactly one of? ParkingLot, ElevatorSystem \(\rightarrow\) Singleton.
\item
  \textbf{Find the family:} Vehicles, Players, Strategies \(\rightarrow\) Factory + ABC.
\end{enumerate}

\subsubsection{The five questions interviewers ask at the end}\label{the-five-questions-interviewers-ask-at-the-end}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\tightlist
\item
  "How would you make this thread-safe?" \(\rightarrow\) Locks per shared resource; mention \texttt{threading.Lock} / \texttt{concurrent.futures}.
\item
  "How would you scale this to 1M users?" \(\rightarrow\) Move to Redis/DB-backed storage; horizontal sharding; mention CAP theorem for rate limiter.
\item
  "Add feature X --- what changes?" \(\rightarrow\) Should be "add a new class, zero changes to existing" (OCP). If it isn\textquotesingle t, your design has a seam problem.
\item
  "How would you test this?" \(\rightarrow\) Unit test each strategy independently; mock dependencies; inject test doubles (e.g., moves iterator in TicTacToe).
\item
  "What did you deprioritize?" \(\rightarrow\) Show you know the tradeoffs: "I skipped persistence, thread-safety on the outer loop, and error telemetry. For production I\textquotesingle d add X."
\end{enumerate}

\textbf{Final interview takeaway: In a 45-min round, a clean, running 200-line solution with 4 well-named classes beats a 500-line "enterprise" solution with 12 interfaces that doesn\textquotesingle t run. Write the smallest correct design, name the patterns you used, and show one extensibility path. That\textquotesingle s the formula.}
